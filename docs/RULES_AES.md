# AES (Agentic Engineering System) Rules

Rules enforced by the `auto-lint` architecture checker on layer boundaries and code structure.

> **Catalog note**: This document defines 31 rule codes spanning AES001–AES033. Codes **AES028 and AES029 are reserved** for future taxonomy rules and intentionally have no entry in the table below.

| code   | name                         | message                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| :----- | :--------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| AES001 | import-layer-violation       | Statement: Layer '{layer}' cannot import from '{target}'.`<br>`WHY? Cross-layer leakage breaks architectural decoupling and creates circular dependencies.`<br>`FIX: Use ports/protocols from the contract layer instead of direct outer-layer imports.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| AES002 | mandatory-import-missing     | Statement: Layer '{layer}' must import from {layers}.`<br>`WHY? Mandatory dependencies ensure that domain-driven contracts are enforced at compile-time.`<br>`FIX: Import the required taxonomy or contract modules immediately.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| AES003 | naming-convention            | Statement: Filename does not follow word1_word2_word3.py pattern.`<br>`WHY? Descriptive, multi-word filenames are required to prevent ambiguity in a layered system.`<br>`FIX: Rename the file using at least two underscores (e.g., user_vo.py).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| AES004 | file-too-large               | Statement: File exceeds {max} lines.`<br>`WHY? Large files violate SRP and are difficult to maintain or test.`<br>`FIX: Split the module into smaller, focused files.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| AES005 | file-too-short               | Statement: File is less than {min} lines.`<br>`WHY? Small files clutter the directory; logic should be merged into a parent module.`<br>`FIX: Merge this logic into a related module or expand it.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| AES006 | primitive-usage              | Statement: Primitive '{type}' used in class attributes, function arguments, or return types in domain boundaries.`<br>`WHY? Core domain integrity requires the use of Value Objects in contract, taxonomy entities, taxonomy errors, and taxonomy events, while adapter layers (infrastructure, capabilities, surfaces) may use primitives as supporting types.`<br>`FIX: Use domain Value Objects instead of raw primitives in public contracts and domain taxonomy.                                                                                                                                                                                                                                                                                                                                                            |
| AES007 | contract-barrel-violation    | Statement: Contract imports must be from the layer's barrel.`<br>`WHY? Direct internal imports bypass the layer's public API boundary.`<br>`FIX: Change the import to target 'contract' instead of 'contract.internal_module'.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| AES008 | contract-suffix-mismatch     | Statement: Contract file missing _port, _protocol, or _io suffix.`<br>`WHY? | AES009 | mandatory-class-definition   | Statement: File is missing a class definition.`<br>`WHY? Logic must be encapsulated in classes to inherit from contracts (ports/protocols/io).`<br>`FIX: Wrap your logic inside a proper class that implements its corresponding contract.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| AES010 | forbidden-suffix             | Statement: File suffix is explicitly forbidden for this layer.`<br>`WHY? Forbidden suffixes (like _vo in infrastructure) indicate a layer responsibility violation.`<br>`FIX: Move the code to the correct layer or rename the file.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| AES011 | suffix-mismatch              | Statement: File suffix does not match allowed patterns for this layer.`<br>`WHY? Strict suffixes are required to communicate the role of components (e.g., _analyzer, _adapter).`<br>`FIX: Rename the file to match one of the allowed layer suffixes.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

| AES013 | internal-all-forbidden       | Statement:__all__ detected in a non-init file.`<br>`WHY? | AES014 | bypass-comment-violation     | Statement: Forbidden bypass comments (noqa, type: ignore) detected.`<br>`WHY? | AES017 | orphan-code-detection        | Statement: File is unreachable from surface entry points.`<br>`WHY? | AES018 | surface-hierarchy-violation  | Statement: Surface file is not imported from its layer barrel.`<br>`WHY? | AES019 | passive-surface-violation    | Statement: Surface file contains complex domain logic instead of acting as thin pass-through.`<br>`WHY? Keep surfaces declarative.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| AES020 | circular-import-violation    | Statement: Circular import detected between modules.`<br>`WHY? | AES021 | agent-role-violation         | Statement: File violates the specific behavioral mandate of its assigned role (e.g., statelessness, coordination).`<br>`WHY? | AES022 | surface-role-violation       | Statement: Complex domain logic or state detected in a passive surface layer.`<br>`WHY? Surfaces must only handle I/O and delegation, acting as a pass-through to agents.`<br>`FIX: Move business logic to the agent layer and keep the surface layer passive/declarative.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| AES023 | surface-dependency-violation | Statement: Surface file imports from forbidden layers (infrastructure, capabilities).`<br>`WHY? | AES024 | agent-any-bypass             | Statement:`Any` type annotation detected in agent layer.`<br>`WHY? `Any` disables mypy type checking entirely — every method call on an `Any`-typed reference bypasses verification, hiding missing methods, wrong argument types, and missing VO construction.`<br>`FIX: Replace `Any` with a typed container protocol/interface in the DI container.                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| AES025 | mcp-tool-schema-violation    | Statement: MCP tool missing docstring, untyped parameters, or invalid JSON Schema.`<br>`WHY? MCP tools must declare valid JSON Schema so clients can validate input before tool calls. The docstring becomes the tool description in tools/list.`<br>`FIX: Add a docstring, type all parameters with Pydantic models or primitives, and ensure schema keywords are valid JSON Schema types.                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| AES026 | forbidden-inheritance        | Statement: Class inherits from a forbidden port or protocol.`<br>`WHY? Restricting inheritance prevents architectural leakage and ensures layers communicate via clean boundaries.`<br>`FIX: Remove the forbidden inheritance base or change the component to use composition/aggregation.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| AES027 | mandatory-inheritance        | Statement: File imports contract but no class inherits from any of them.`<br>`WHY? Layers like agent, capabilities, and infrastructure must implement their imported contract ports/protocols via inheritance.`<br>`FIX: Make at least one class in this file inherit from one of the imported contracts.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| AES030 | capability-method-not-found  | Statement: Capability method referenced in COMMAND_CATALOG does not exist on the target class.`<br>`WHY? Mismatched method names between dispatch catalog and capability class cause runtime dispatch failures.`<br>`FIX: Sync the method name in COMMAND_CATALOG with the actual method defined on the capability class.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| AES031 | single-capability-bottleneck | Statement: All dispatch routes go to a single capability class when multiple specialized options exist.`<br>`WHY? Dispatch should distribute actions across capabilities by concern — a single class handling everything indicates under-utilization of the architecture.`<br>`FIX: Create or use specialized capability classes for distinct action types.                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| AES032 | missing-vo-construction      | Statement: Capability method call missing required request/data Value Object parameter.`<br>`WHY? Capability methods expect typed VO arguments — calling without them bypasses domain validation and creates runtime fragility.`<br>`FIX: Pass the appropriate request or data VO to every capability method call.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| AES033 | constant-purity              | Statement: Taxonomy `_constant` file contains non-constant declarations (`struct`, `enum`, `fn`, `impl`, `mod`, `pub fn`, `pub struct`, `pub enum`, `pub mod`, `pub use`).`<br>`WHY? Constants are the only Taxonomy role permitted to expose raw primitives (AES006 exception) because constants are primitives by definition. Mixing in `struct`/`enum`/`fn`/`impl` collapses the role boundary and reintroduces primitive leakage that `_vo` already handles — destroying the type/value separation and producing files that are neither constant tables nor proper VOs.`<br>`FIX: Move the non-constant declarations to the appropriate file (a `_vo`, `_entity`, or capability module) and keep the `_constant` file restricted to `pub const` / `pub static` declarations only. |

## Multi-Condition Rules

Sembilan AES code memiliki **lebih dari satu violation message** yang berbeda-beda tergantung layer/scope pelanggaran. Berikut daftar lengkap setiap variant pesan per kode.

### AES001 — IMPORT_LAYER_VIOLATION (8 variants)

| Scope                                            | Message                                                                                                                                                                                                                                                                                                                                      |
| ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `taxonomy(vo)`                                 | **AES001 IMPORT_LAYER_VIOLATION: Taxonomy VO imported from an outer layer.**`<br>`WHY? Taxonomy is the system's foundation and must have zero dependencies on outer layers.`<br>`FIX: Remove all imports to agent, infrastructure, surfaces, contract, or capabilities layers.                                                     |
| `taxonomy(entity, error, event)`               | **AES001 IMPORT_LAYER_VIOLATION: Taxonomy imported from an outer layer.**`<br>`WHY? Taxonomy is the bottom-most layer and must not depend on outer system layers.`<br>`FIX: Remove all imports to agent, infrastructure, contract, or capabilities layers.                                                                         |
| `contract(port, protocol)`                     | **AES001 IMPORT_LAYER_VIOLATION: Pure contract (_port/_protocol) depends on execution logic.**`<br>`WHY? Ports and Protocols define logic shapes and must not know about orchestration or implementation.`<br>`FIX: Remove all imports except from taxonomy.                                                                       |
| `contract(aggregate)`                          | **AES001 IMPORT_LAYER_VIOLATION: Contract Aggregate depends on implementation layers.**`<br>`WHY? The Aggregate represents the system's formal promise and must remain agnostic to implementation details.`<br>`FIX: Remove all imports from infrastructure, capabilities, agent, or surfaces layers.                              |
| `capabilities`                                 | **AES001 IMPORT_LAYER_VIOLATION: Capability imported from an unauthorized layer or sibling module.**`<br>`WHY? Capabilities must be independent; importing siblings or outer layers causes circular cycles.`<br>`FIX: Remove imports from other capabilities, infrastructure, surfaces, or agent layers. Use Dependency Injection. |
| `infrastructure`                               | **AES001 IMPORT_LAYER_VIOLATION: Infrastructure component imported from an unauthorized layer or sibling.**`<br>`WHY? Infrastructure adapters must be isolated; importing siblings creates technical coupling and cycles.`<br>`FIX: Remove imports from other infrastructure, surfaces, capabilities, or agent modules.            |
| `agent(orchestrator, coordinator, dispatcher)` | **AES001 IMPORT_LAYER_VIOLATION: Orchestrator depends on a sibling or governance component.**`<br>`WHY? Orchestrators must only use ServiceContainerAggregate (contract) to avoid circular dependencies.`<br>`FIX: Use self.container.get(ContractAggregate) and only import from 'contract' and 'taxonomy' layers.                |
| `agent(manager, handler, result, state)`       | **AES001 IMPORT_LAYER_VIOLATION: Support module (leaf node) depends on orchestration or outer layers.**`<br>`WHY? Manager/Result must be independent of the wiring and logic flow to prevent cycles.`<br>`FIX: Remove all Agent/Infra/Cap imports. Stick to Taxonomy and Contract Aggregate.                                       |
