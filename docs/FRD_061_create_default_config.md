## 2. Introduction
### 2.1 Purpose
This document defines the create default config CLI command `setup init`. It generates a `lint_arwaky.config.<language>.yaml` file in the target project directory with sensible defaults, enabling developers to quickly bootstrap architecture rule configuration for their project.

### 2.2 Scope
**In-Scope:**
- `lint-arwaky-cli setup init` — create default config in current directory
- `lint-arwaky-cli setup init --language rust` — specify language explicitly
- Language auto-detection from project structure
- Interactive language selection when ambiguous
- Pre-populated with all 31 AES rules with default severity levels
- Customizable threshold, exclusions, and layer definitions

**Out-of-Scope:**
- Modifying existing config files
- MCP client configuration (handled by FR-062)
- Environment diagnostics (handled by FR-060)

### 2.3 Glossary
| Term | Definition |
|------|------------|
| **Default config** | Pre-populated `lint_arwaky.config.*.yaml` with all rules at default severity |
| **Language detection** | Auto-detection from `src-{lang}/` dirs or manifest files |
| **Rule entry** | YAML entry defining an AES rule: code, severity, enabled, options |

## 3. Feature Overview
### 3.1 Background & Problem
Setting up Lint Arwaky for a new project required manually creating a config YAML file with the correct schema. There was no template or generator, so developers had to reference documentation to write the config from scratch.

### 3.2 Business Goals
- Reduce setup time for new projects to seconds
- Provide a complete, working config with all 31 rules
- Support all three target languages
- Generate language-appropriate config automatically

### 3.3 Target Users
- **New Users**: Setting up Lint Arwaky for the first time
- **Project Onboarding**: Bootstrap architecture rules for new projects
- **AI Agents**: Auto-configure projects during automated setup

## 4. Functional Requirements
### 4.1 User Stories
- **US-001:** As a new user, I want to run `setup init` in my project directory to get a working config file, so I can start linting immediately.
- **US-002:** As a developer with a Rust project, I want `setup init` to detect my language and generate the correct config file name.
- **US-003:** As an architect, I want the generated config to include all 31 AES rules with sensible defaults, so I can customize from a complete starting point.

### 4.2 Use Cases & Workflow
**Init Pipeline:**
```
lint-arwaky-cli setup init
  │
  ├─► 1. Detect project language
  │     ├── src-rust/ → Rust
  │     ├── src-python/ → Python
  │     ├── src-javascript/ → JavaScript
  │     ├── Cargo.toml → Rust
  │     └── ambiguous → prompt user
  │
  ├─► 2. Check for existing config
  │     ├── lint_arwaky.config.rust.yaml exists?
  │     └── If yes → warn and ask to overwrite
  │
  ├─► 3. Generate default config
  │     ├── All 31 AES rules at default severity
  │     ├── Language-specific layer definitions
  │     ├── Default threshold: 70
  │     └── File patterns for source scanning
  │
  └─► 4. Write lint_arwaky.config.<language>.yaml
        └── Report: "Config created at ./lint_arwaky.config.rust.yaml"
```

**Generated Config Example:**
```yaml
# Lint Arwaky Configuration — Auto-generated by `setup init`
project:
  language: rust
  name: my-project
  version: 1.0.0

ci:
  threshold: 70

rules:
  aes001:
    enabled: true
    severity: CRITICAL
  aes002:
    enabled: true
    severity: HIGH
  aes003:
    enabled: true
    severity: LOW
  # ... all 31 rules ...

layers:
  taxonomy:
    prefixes: ["taxonomy_"]
    suffixes: ["_vo", "_entity", "_event", "_error", "_constant"]
  contract:
    prefixes: ["contract_"]
    suffixes: ["_port", "_protocol", "_aggregate"]
  capabilities:
    prefixes: ["capabilities_"]
    suffixes: ["_checker", "_analyzer", "_processor"]
  infrastructure:
    prefixes: ["infrastructure_"]
    suffixes: ["_adapter", "_provider", "_scanner"]
  agent:
    prefixes: ["agent_"]
    suffixes: ["_container", "_orchestrator", "_coordinator", "_registry", "_manager"]
  surface:
    prefixes: ["surface_"]
    suffixes: ["_command", "_handler", "_controller"]

source:
  include:
    - "src-rust/**/*.rs"
  exclude:
    - "**/target/**"
    - "**/vendor/**"
```

### 4.3 Business Rules
- Config file name: `lint_arwaky.config.{language}.yaml`
- If config file already exists, prompt for overwrite confirmation
- Language auto-detection: same 4-tier system as FR-002
- If language is ambiguous, prompt user with options
- Generated config includes ALL 31 AES rules with FRD-defined default severities

## 8. Empirical Findings (Code Audit)

### 8.1 Current Implementation

| Component | Location | Lines | Status |
|-----------|----------|-------|--------|
| CLI setup init command | `project-setup/surface_init_command.rs` | — | **FULLY IMPLEMENTED** |
| Config generator | `project-setup/capabilities_config_generator.rs` | — | **FULLY IMPLEMENTED** — template-based generation |
| Language detection reuse | `config-system/infrastructure_detector_provider.rs` | 40 | **FULLY IMPLEMENTED** |
| Config writer | `project-setup/infrastructure_config_writer.rs` | — | **FULLY IMPLEMENTED** — writes YAML to disk |

### 8.2 Bugs Found

1. **Config generator hardcodes Rust layer definitions** — Python and JavaScript configs get Rust-specific suffixes and prefixes
   - **Impact**: Python/JS projects get incorrect layer configuration
   - **Fix**: Language-specific template selection

2. **No overwrite confirmation in non-interactive mode** — when stdin is not a TTY (CI mode), setup init silently overwrites existing config
   - **Impact**: CI pipelines may accidentally destroy customized config
   - **Fix**: Skip overwrite in non-interactive mode unless `--force` flag provided

3. **Generator does not respect `--language` flag in all cases** — the `--language` flag is parsed but language detection still runs and takes priority
   - **Impact**: `--language python` on a Rust project still generates Rust config
   - **Fix**: Explicit `--language` flag should override auto-detection

### 8.3 What Needs to Be Added

- **Language-specific templates**: Separate YAML templates for Rust, Python, JavaScript
- **--force flag**: Allow overwrite in non-interactive mode
- **--language override**: Fix flag priority over auto-detection

### 8.4 What to Keep

- **Full rule set** ✅ — all 31 AES rules included in generated config
- **Layer definitions** ✅ — complete 6-layer architecture template
- **CI threshold default** ✅ — sensible 70 default
- **Config YAML validity** ✅ — output is valid YAML matching parser schema

### 8.5 Empirical Evidence from Test Projects

- `lint-arwaky-cli setup init` generates a valid config that passes `check .` loading
- Config contains all 31 AES rules with correct codes
- Pending Review: Language-specific templates, --language flag fix
